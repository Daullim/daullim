# ADR-004 FE 세부 — 라우팅·상태관리·지도·차트

## 상태
부분 승인 (2026-07-20, 결정자: 이윤서) — §1 라우팅 확정, §2~4는 **추후 결정(보류)**

## 맥락
- React + Vite 확정(PRD v2). 시연 URL 1개, 계정 2종 역할 분기(M-14), 폴링 5~10s(WBS D10).
- 기능명세서(2026-07-19) 3화면 체계: SCR-01 위험지도 / SCR-02 우선순위 리스트 / SCR-03 순찰 대상 상세.
- 사용자 최우선 축: 빠른 지도·현재 위치 실시간성(단말 자기 위치 표시 범위), 부가 길찾기.
- 도농 전환은 토글이 아니라 지역 선택(FR-001)에 따른 `region_type` 자동 결정 (기능명세서 정본).

## 검토한 선택지
- 라우팅: **단일 앱 + 역할 라우트 분리(채택)** vs 앱 2개 분리 배포(URL 1개 확정사항과 충돌 — 기각)

## 결정
1. **라우팅 (확정):** 단일 React 앱. `/login` → 역할 분기 → 관제 `/control`(SCR-01 지도 + SCR-02 리스트), 현장 `/field`(SCR-02 명단 + SCR-03 상세 회신). `/field`만 모바일 우선 반응형.
2. 이하 **보류 (추후 결정)** — 제안안 기록:
   - 서버 상태·폴링: TanStack Query, `refetchInterval` 7s, 저장 후 invalidate. Redux/Zustand 반려 의견.
   - 지도: react-leaflet + `preferCanvas: true`, GeoJSON 경량화(좌표 5자리·시군구 클리핑)는 pipeline 책임, 현재 위치는 `watchPosition` 파란 점.
   - 차트: 라이브러리 미도입 — 집계 카운터는 숫자 카드로.
   - 길찾기: 카카오맵/티맵 딥링크(외부 API 의존 0 유지).

## 근거
- URL 1개·컴포넌트 재사용·기능명세서 3화면의 자연 배치.

## 결과 (트레이드오프 수용)
- §2~4 보류로 FE 착수 시점(WBS 1주차)에 재결정 필요 — D1(프로젝트 셋업) 전까지 확정 권장. 보류 항목 확정 전 지도·폴링 구현 착수 불가.
