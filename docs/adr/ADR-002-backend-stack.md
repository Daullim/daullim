# ADR-002 BE 스택 — Spring Boot + Java

## 상태
승인 (2026-07-20, 결정자: 이윤서)

## 맥락
- 워크로드 실체: 정적 seed 테이블 CRUD 서빙 7 엔드포인트 + 하드코딩 계정 2종 로그인 + 폴링 조회. 실시간 추론 0(PRD v2 대안①), 공간연산 0(대안④).
- 역할 분담 실측: 이윤서 = BE·인프라·FE·디자인·기획 전담 / 박지현 = BE API 일부·FE 연동 일부 보조 — PRD v2의 "BE 1·FE 1 병렬" 전제와 다름 (WBS 재배치 필요, Step 3에서 제안).
- 사용자 지정 우선 축: 빠른 지도·현재 위치 실시간성 / AI 우선순위 큐 산출 / (부가) 길찾기 — 분석 결과 세 축 모두 BE 스택 중립(지도·위치는 FE, AI 산출은 pipeline Python, 길찾기는 딥링크 권장).

## 검토한 선택지
1. Spring Boot + Kotlin — 본인 실전 이력(BLT·Muses·HAN-Spoon, ktlint 세팅). 주류 포트폴리오.
2. **Spring Boot + Java — 채택.** 채용 시장 최대 수요 스택.
3. Node.js + TypeScript (NestJS) — 기술 적합성 1순위 평가(FE 타입 직접 공유, 경량 배포, 1인 풀스택 총처리량). AI 추천안.
4. FastAPI — pipeline 언어 통일·OpenAPI 자동. FE 비중 증가 국면에서 상대 이점 약화 평가.

## 결정
**Spring Boot + Java.** BE 언어 ≠ seed 언어(Python) 허용 — seed는 pipeline/의 오프라인 배치로 앱 런타임과 분리(PRD v2 대안①), 런타임 pandas 사용 0이므로 문제 없음.

## 근거 (결정자)
- 백엔드 취업 시장 최대 수요 스택(Java-Spring)으로의 포트폴리오 직결.
- 학습곡선 배제 전제에서 결정자가 시장 신호 가치를 기술 적합성보다 우선함.

## 결과 (트레이드오프 수용 내역 — AI 우려 기록)
- **기술 적합성 열위 수용:** 후보 4종 중 배포 풋프린트(JVM 수백 MB)·보일러플레이트 최대. D6 배포에서 무료/저가 티어 메모리 대응(힙 제한 튜닝 또는 상위 티어) 필요.
- **기존 자산 재사용 불가:** Kotlin 이력·ktlint 세팅 미활용. D7에서 Java용 포맷터/린터(Spotless + google-java-format 등) 신규 세팅 필요.
- FE 타입 공유는 네이티브 불가 → D5에서 springdoc(OpenAPI) 기반 계약 운영으로 보완.
- 시연 성공(제약 1)에는 지장 없음 — 워크로드가 단순해 어느 스택이든 3주 내 완성 가능하다는 평가는 유지.
